# 异常体系

## Unchecked Exception与Checked Exception

| 维度 (Dimension) | Checked Exception (受检异常)                                 | Unchecked Exception (非受检异常)                             |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **编译要求**     | **必须**处理。方法内部必须用 `try-catch` 捕获，或者在方法签名上用 `throws` 关键字声明，否则编译报错。 | **无需**强制处理。编译器不检查，方法可以直接抛出，调用方也可以选择不捕获。 |
| **处理方式**     | 通常代表程序运行时**可预见、可恢复**的错误。例如：文件不存在、网络连接失败、数据库不可用。这些错误是外部环境导致的，程序员应该提供备用方案或优雅地向用户报告。 | 通常代表**编程错误**或运行时**不可恢复**的错误。例如：空指针访问、数组越界、传入非法参数。这些是程序员逻辑错误导致的，应通过修改代码来避免，而不是依赖异常处理。对于 `Error`，如内存溢出，程序通常无法恢复。 |
| **设计目的**     | **强制暴露**潜在的风险点，迫使程序员在编码阶段就考虑异常处理逻辑，从而**提高程序的健壮性和可靠性**。 | **简化代码**，让程序员专注于业务逻辑。将异常处理的精力集中在那些真正需要关注的、由逻辑错误引起的异常上。同时，对于 `Error`，它表示系统级的严重问题，应用程序通常无法处理。 |
